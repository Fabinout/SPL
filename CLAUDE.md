## Project Overview

This repository contains the specification for **SPL (Structured Parenthesized Language)** ‚Äî a low-level, stack-based virtual machine language designed to be easily generated by LLMs and trivially parsed. The full spec is in `SPL-spec-fr.md` (v1.0, written in French).

## Commands

```bash
# Assemble .spl to .rom bytecode
python vm-python/spl_asm.py <input.spl> [output.rom]

# Run a .rom in the VM
python vm-python/spl_vm.py <program.rom>

# Run all tests
python vm-python/tests/run_tests.py
```

## Architecture

- **`vm-python/spl_asm.py`** ‚Äî Two-pass assembler. Pass 1 collects labels and byte offsets. Pass 2 emits bytecode with resolved addresses. Validates argument types (rejects labels for `load`/`store`, range-checks values/ports/addresses).
- **`vm-python/spl_vm.py`** ‚Äî Bytecode interpreter. Harvard architecture: code (ROM) and data (64 KiB RAM) are separate address spaces. Implements console I/O, RNG, 32-bit timer with latch, and SYSCTL_CAPS. Conditional jumps (`jump-if-zero`, `jump-if-not-zero`) pop the tested value.

## SPL Language Quick Reference

### Execution Model
- **Stack-based VM** with a work stack (min 256 entries), a return stack (min 64 entries), and 64 KiB linear memory
- All values are **8-bit unsigned** (mod 256); bytecode addresses are **big-endian**
- Halt on: `(halt)`, PC past end of bytecode, stack underflow/overflow, OOB memory access
- Division by zero returns 0

### Instruction Set
- **Halt**: `halt` (0x00)
- **Stack**: `push` (0x01), `drop`, `dup`, `swap`, `over`
- **Arithmetic/Logic**: `add`, `sub`, `mul`, `div`, `mod`, `and`, `or`, `xor`, `not` (bitwise)
- **Memory**: `load` (0x20), `store` (0x21) ‚Äî direct addressing only, numeric 16-bit constants
- **Control Flow**: `jump`, `jump-if-zero`, `jump-if-not-zero`, `call`, `return` ‚Äî accept label names or literal addresses
- **I/O**: `in` (0x40), `out` (0x41) ‚Äî 8-bit port number
- **Pseudo**: `label` ‚Äî emits no bytes, resolves to byte offset
- Binary operand order: second-from-top is left, top is right

### Port-Mapped I/O (currently implemented in vm-python)
- `0x01`‚Äì`0x03`: Console (data W, status R, flush W)
- `0x10`: RNG (R)
- `0x11`‚Äì`0x14`: Timer 32-bit ms (R, latch on B3 read)
- `0xFF`: SYSCTL_CAPS (R)

### Implementation Notes
- Labels resolve to 16-bit byte offsets; forward references allowed; names must be unique
- `push` values are strictly 0‚Äì255
- Console buffers output, flushes on `\n` or `(out 0x03)`
- VM currently supports Core I/O profile only. Keyboard, video, audio, and mouse ports return 0 / are silently ignored

## Build Artifacts & Cleanup

### `.rom` Files (Bytecode)
**IMPORTANT:** `.rom` files are **generated build artifacts** and should **NEVER** be committed.

- Generated by: `python3 vm-python/spl_asm.py <program.spl>`
- Purpose: Compiled bytecode for the VM
- Lifecycle: Generate ‚Üí Test ‚Üí **Delete** before committing

### Cleanup Procedure
After testing a new program:
```bash
# After assembly & testing
python3 vm-python/spl_asm.py examples/poem.spl -o poem.rom
python3 vm-python/spl_vm.py examples/poem.rom
# ... verify it works ...

# BEFORE COMMITTING: Delete the .rom file
rm examples/poem.rom

# Commit only the .spl source
git add examples/poem.spl
git commit -m "Add poem example"
```

### Gitignore
The `.gitignore` file already excludes:
- `*.rom` ‚Äî bytecode files
- `__pycache__/` ‚Äî Python cache
- `.pytest_cache/` ‚Äî pytest cache
- IDE files (`.vscode/`, `.idea/`)
- OS files (`.DS_Store`, `Thumbs.db`)

If you accidentally commit a `.rom` file, remove it:
```bash
git rm --cached examples/poem.rom
git commit -m "Remove accidentally committed .rom file"
```

### Why Not Commit .rom Files?
1. **Size** ‚Äî Bytecode bloats the repository
2. **Regenerability** ‚Äî .rom files can always be recreated from .spl
3. **Redundancy** ‚Äî Source of truth is the .spl file
4. **Clean history** ‚Äî Easier to review code changes in .spl files
5. **Merge conflicts** ‚Äî Binary files can't be merged meaningfully
